# 思考题回答

## 问题1：在本练习中，货架模板的"内部状态"和"外部状态"分别是什么？为什么要分离这两种状态？

**内部状态**：货架模板的固定规格属性，包括类型（横梁式、驶入式等）、长、宽、高、承重、可放层数。这些属性对所有使用该模板的货架都是相同的，不会随使用场景变化。

**外部状态**：每个具体货架独有的动态信息，包括货架编号（如 S-A-001）、所在库区（如 A区、B区、C区）、占用状态等。这些信息因具体货架而异，会随业务变化。

**分离原因**：
1. **内存优化**：如果不分离，1000个横梁式货架会各自存储一份完全相同的规格数据（类型、长宽高等），造成大量内存浪费。分离后，1000个货架共享1个模板对象，仅各自保存不同的编号、库区等外部状态。
2. **数据一致性**：模板数据集中管理，如果需要修改某类货架的规格（如承重从500kg调整为600kg），只需修改模板对象，所有引用该模板的货架自动生效。
3. **符合享元模式核心思想**：将不变的部分（内部状态）提取为共享的享元对象，将变化的部分（外部状态）作为参数传入，实现了对象的轻量化复用。

## 问题2：装饰器模式中，抽象装饰器为什么要继承抽象构件？如果不继承，会有什么问题？

**为什么要继承**：
抽象装饰器（如 `PickingDecorator`）继承抽象构件（如 `PickingStrategy`），是为了保证装饰器本身也是构件的一种，使得装饰后的对象和原始对象具有相同的类型和接口。这样客户端代码无需区分处理的是基础构件还是被装饰过的构件，实现了透明装饰。

**如果不继承会产生的问题**：
1. **类型不兼容**：装饰器无法替代原始构件的位置。方法签名如果要求接收 `PickingStrategy` 类型参数，传入装饰器会编译报错，无法实现灵活组合。
2. **无法嵌套装饰**：装饰器无法包装另一个装饰器，因为类型不匹配。例如 `ColdChainPickingDecorator` 构造函数需要接收 `PickingStrategy`，如果 `WavePickingDecorator` 不是 `PickingStrategy` 的子类，就无法实现 `ColdChain(Wave(Normal))` 这样的多层嵌套。
3. **违反里氏替换原则**：客户端需要用 `if-else` 判断当前是基础构件还是装饰器，增加了耦合度，违背了开闭原则。

## 问题3：对比"用继承实现拣货策略"和"用装饰器模式实现拣货策略"，两种方式的优缺点分别是什么？

### 继承方式

如果用继承实现，需要为每种策略组合定义一个子类：

```
NormalPicking
WavePicking extends NormalPicking
ColdChainPicking extends NormalPicking
FragilePicking extends NormalPicking
WaveColdChainPicking extends NormalPicking
WaveFragilePicking extends NormalPicking
ColdChainFragilePicking extends NormalPicking
WaveColdChainFragilePicking extends NormalPicking
```

**优点**：
- 类层次关系清晰直观
- 每个子类的行为确定，不含运行时变化

**缺点**：
- **类爆炸问题**：n个规则会产生 2^n 个子类。3个规则需要7个子类，新增第4个规则则翻倍为15个。
- **静态绑定**：策略在编译时确定，无法在运行时动态组合或增减规则。
- **代码重复**：大量子类之间存在相似的组合逻辑，维护困难。
- **违反开闭原则**：每新增一个规则组合都要新建子类，修改已有代码的风险高。

### 装饰器模式方式

**优点**：
- **避免类爆炸**：n个规则只需要 n+1 个类（1个基础构件 + n个装饰器），组合在运行时动态构建。
- **动态组合**：可以在运行时任意组合、增减装饰器，如 `new ColdChain(new Wave(new Normal()))` 灵活拼接。
- **符合开闭原则**：新增规则只需添加一个装饰器类，无需修改已有代码。
- **职责单一**：每个装饰器只负责一个规则，代码简洁易维护。

**缺点**：
- **调试复杂**：多层装饰器嵌套后，调用栈较深，跟踪执行流程不如继承直观。
- **顺序敏感**：装饰器的叠加顺序可能影响结果，需要调用方自行控制（通过优先级机制可缓解）。
- **无法方便地移除中间层**：一旦装饰后，不方便去除某一层装饰（需要重建整个链）。
